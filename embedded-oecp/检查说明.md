## 一、源码认证（3 项）

### 1.1 内核（KernelChecker）

**检查目标**：验证内核源码来源合规、版本在主流范围内，且与目标设备一致。

**检查步骤**（5 步）：

| 步骤 | 检查内容 | 判定标准 |
|------|----------|----------|
| Step 1 | 从 `manifest.yaml` 获取内核源码仓库地址和版本号 | 成功获取 manifest 条目（包含 `remote_url` 和 `version`） |
| Step 2 | 内核托管仓库在 openEuler 组织下 | `remote_url` 包含 `atomgit.com/openeuler` 或 `gitee.com/openeuler` |
| Step 3 | manifest.yaml 信息与构建工程内核源码一致 | 本地 git upstream URL 与 manifest remote 一致，且 manifest 版本号在远端仓库存在 |
| Step 4 | 内核版本在 oee 主流范围内 | 从 Makefile 解析版本号，major.minor（如 5.10、6.6）在配置的 `mainstream_versions` 列表中 |
| Step 5 | 目标设备内核版本在主流范围内 | SSH 到目标设备执行 `uname -r`，major.minor 在 `mainstream_versions` 列表中 |

**主流版本范围**：`["5.10", "6.6"]`（可通过 `config.yaml` 的 `kernel.mainstream_versions` 配置）

**证据文件**：
- `kernel_step1_manifest_info.png` — manifest 信息截图
- `kernel_step2_repo_page.png` — 仓库页面截图
- `kernel_step3_local_verify.png` — 本地 git 验证截图
- `kernel_step4_version_check.png` — 版本检查截图
- `kernel_step5_target_uname.png` — 目标设备 uname 截图

**数据来源**：
- 构建环境：`oebuild bitbake-getvar MACHINE`、`tmp/work-shared/<MACHINE>/kernel-source`、Makefile
- 基线：`manifest.yaml`（`<source_dir>/yocto-meta-openeuler/.oebuild/manifest.yaml`）
- 目标设备：SSH `uname -r`

### 1.2 基础中间件（MiddlewareChecker）

**检查目标**：验证目标设备的 C 库（glibc/libc.so）与 openEuler 官方基线一致。

**检查步骤**（3 步）：

| 步骤 | 检查内容 | 判定标准 |
|------|----------|----------|
| Step 1 | 从目标设备提取 libc.so 的 MD5 值 | SSH 到设备执行 `md5sum /usr/lib*/libc.so* /lib*/libc.so*`，成功获取 MD5 |
| Step 2 | 从 env.yaml 获取对应架构的基线 MD5 | `env.yaml` 中 `runtime_md5.<arch>.libc_so` 条目存在 |
| Step 3 | 对比设备 MD5 与基线 MD5 | 两者完全一致（严格相等） |

**判定标准**：MD5 严格相等，无容差。

**证据文件**：
- `middleware_step1_device_md5.png` — 设备 libc.so MD5 截图
- `middleware_step2_baseline.png` — env.yaml 基线截图
- `middleware_step3_compare.png` — 比对结果截图

**数据来源**：
- 目标设备：SSH `md5sum`
- 基线：`env.yaml`（从 `baseline_repo` + `baseline_branch` 远端获取）

---

### 1.3 其他软件包（PackageChecker）

**检查目标**：验证镜像依赖包的社区来源占比达到要求。

**检查流程**：

1. 从构建环境生成 `pn-buildlist`（`oebuild bitbake -g openeuler-image`）
2. 过滤掉 `-native`、`-cross`、`-crosssdk`、`-initial`、`-canonical`、`-nativesdk` 后缀的包
3. 对每个包查询 `OPENEULER_LOCAL_NAME`，在 `src/openeuler/<local_name>` 下获取 git remote URL 和 HEAD commit
4. 判定是否社区包：remote URL 包含 `atomgit.com/src-openeuler` 或 `gitee.com/src-openeuler` 等 openEuler 组织
5. 判定是否基线匹配：HEAD commit 与 `manifest.yaml` 中记录的 version 一致
6. 计算社区包占比

**判定标准**：社区包占比 **≥ 70%**（`COMMUNITY_RATIO_THRESHOLD = 0.70`）

```
社区包 = (remote_url 属于 openEuler 组织) AND (commit == manifest version)
占比 = 社区包数 / 总包数
```

**证据文件**：
- `package_dependency_list.xlsx` — 依赖包明细 Excel（两个 sheet：镜像依赖包列表 + 社区基线 Manifest）

## 二、构建认证（3 项）

### 2.1 编译链（CompilerChecker）

**检查目标**：验证构建环境使用的 gcc 和 glibc 与 openEuler 官方编译链一致。

**检查步骤**（3 步）：

| 步骤 | 检查内容 | 判定标准 |
|------|----------|----------|
| Step 1 | 从远端 env.yaml 获取 gcc 和 glibc 的 MD5 基线 | `toolchain_md5.<arch>.gcc` 和 `runtime_md5.<arch>.libc_so` 条目均存在 |
| Step 2 | 计算构建环境中 gcc 和 glibc 文件的 MD5 | 工具链目录下所有目标文件均可定位并成功计算 MD5 |
| Step 3 | 比对实际 MD5 与基线 MD5 | gcc 和 glibc 的 MD5 均**完全一致** |

**判定标准**：gcc 和 glibc 的 MD5 均**严格相等**，两者必须同时通过。

**证据文件**：
- `compiler_step1_env_yaml.png` — env.yaml 基线截图
- `compiler_step2_md5sum.png` — 构建环境 MD5 计算截图
- `compiler_step3_conclusion.png` — 比对结论截图

**数据来源**：
- 基线：`env.yaml`（远端 `baseline_repo` + `baseline_branch`）
- 构建环境：`toolchain.dir` 配置或根据架构自动推导

---

### 2.2 构建工程（ProjectChecker）

**检查目标**：验证构建工程目录的命令可用性和结构完整性。

**检查步骤**（2 步）：

| 步骤 | 检查内容 | 判定标准 |
|------|----------|----------|
| Step 1 | 构建命令 oebuild bitbake 可用 | 执行 `oebuild bitbake --version`，输出包含 "BitBake Build Tool Core version" |
| Step 2 | 构建目录结构标准 | 所有必需的目录和文件均存在（见下表） |

**必需目录**：`cache`、`conf`、`output`、`tmp`

**必需配置文件**：`conf/bblayers.conf`、`conf/local.conf`

**必需根文件**：`compile.yaml`、`.env`、`oebuild.log`

**判定标准**：所有目录和文件均存在（全部通过才算 PASS）。

**证据文件**：
- `project_step1_build_command.png` — bitbake 版本输出截图
- `project_step2_build_structure.png` — 目录结构检查截图（逐项 ✓/✗）

### 2.3 包列表（PkgListChecker）

**检查目标**：验证镜像 manifest 中软件包与社区基线的版本一致性。

**三层过滤流程**：

```
镜像 manifest（如 2242 包）
  │
  ├─ ① 排除 packagegroup 包
  │
  ├─ ② 判断是否社区包（在 manifest.yaml 基线中查找）
  │    ├─ 在基线中 → 社区包 → 从 src/openeuler/ 获取 remote，比对版本一致性
  │    └─ 不在基线 → 非社区包
  │
  └─ ③ 排除辅助配方（PV=1.0 且 SRC_URI 无压缩包）
```

**关键技术点**：

- **PKG 反向映射**：Yocto 会重命名包（如 `libnl` → `libnl-3-200`），通过 pkgdata 的 `PKG:` 字段建立反向映射
- **pkgdata 直接读取**：从 `tmp/pkgdata/<MACHINE>/runtime/` 直接读取 `OPENEULER_LOCAL_NAME`、`PN`、`PV`
- **辅助配方识别**：PV=1.0 且 SRC_URI 为空或仅含 `file://`（无 `.tar.*`、`.zip` 等压缩包），通过 `oebuild bitbake -e <pn>` 获取 SRC_URI
- **基线获取**：从 `baseline_repo`（`baseline_branch`）远程获取 `.oebuild/manifest.yaml`

**判定标准**：社区包版本一致率 **≥ 70%**（`PKG_CONSISTENCY_THRESHOLD = 0.70`）

```
一致率 = 社区包版本一致数 / (总包数 - packagegroup包数 - 辅助配方数)
```

分母包含：社区包一致 + 社区包不一致 + 非社区包 + 社区包无源码目录

**证据文件**：
- `package_dependency_list.xlsx` — 包列表检查结果 Excel（含软件包名、版本、一致性状态、社区包 remote/version）
- `pkglist_step1_consistency.png` — 一致性检查结果截图（含进度条和百分比）

## 三、运行时认证（3 项）

### 3.1 C 库运行时（MiddlewareRuntimeChecker）

**检查目标**：验证目标设备上 glibc 的 ABI 兼容性。

**检查步骤**（2 步）：

| 步骤 | 检查内容 | 判定标准 |
|------|----------|----------|
| Step 1 | 上传 ABI 检测程序并验证 MD5 | 将 `glib-abi-check-<arch>` 上传到目标设备，MD5 与官方 `md5sums` 文件记录一致 |
| Step 2 | 在目标设备运行 ABI 检测程序 | 程序退出码为 0（运行成功） |

**检测程序来源**：`src/yocto-meta-openeuler/.oebuild/glib2.0-abi-check/`（按架构选择二进制）

**架构映射**：

| 架构 | 检测程序 |
|------|----------|
| aarch64 | `glib-abi-check-aarch64` |
| arm32 | `glib-abi-check-arm32` |
| riscv64 | `glib-abi-check-riscv64` |
| x86_64 | `glib-abi-check-x86` |

**判定标准**：Step 1 MD5 严格相等 + Step 2 退出码为 0，两步均通过才算 PASS。

**证据文件**：
- `libc_runtime_step1a_official_md5.png` — 官方 MD5 截图
- `libc_runtime_step1b_device_md5.png` — 设备 MD5 及比对截图
- `libc_runtime_step2_abi_run.png` — ABI 检测程序运行结果截图

---

### 3.2 基础功能 AT（ATTestChecker）

**检查目标**：通过 mugen 框架执行社区 AT（Acceptance Test）测试用例，验证系统基础功能。

**检查流程**：

1. 克隆 mugen 框架（`mugen.remote`，默认 `https://atomgit.com/openeuler/mugen.git`）
2. 复制 `embedded_*.json` suite2cases 文件
3. 安装 SDK（如配置了 `sdk_file`）
4. 根据模板 `embedded_osv_at.json` 生成 mugen 配置（填充设备 IP、密码等）
5. 执行 `bash combination.sh -f embedded_osv_at.json -r`
6. 解析 `combination_results/` 下的 `succeed/` 和 `failed/` 目录统计结果

**判定标准**：**100% 通过率**（`passed == total`，所有 AT 用例必须全部成功）

**证据文件**：
- `at_test_execute_log.txt` — 完整执行日志（stdout/stderr + 退出码）
- `at_test_step1_execute.png` — 执行输出截图（最后 15 行，彩色）
- `at_test_step2_results.png` — 结果汇总截图（总数/通过/失败/通过率 + 失败用例列表）

---

### 3.3 POSIX（PosixChecker）

**检查目标**：通过 mugen 框架执行 POSIX 测试用例，验证 POSIX 兼容性。

**检查流程**：

与 AT 测试流程基本一致，区别在于：

- 使用 `embedded_osv_posix.json` 模板
- 解析结果时仅统计目录名包含 `posix` 的 combination 结果
- 失败时的修复建议为"请检查 glibc 及内核兼容性"

**判定标准**：**100% 通过率**（`passed == total`，所有 POSIX 用例必须全部成功）

**证据文件**：
- `posix_execute_log.txt` — 完整执行日志
- `posix_step1_execute.png` — 执行输出截图
- `posix_step2_results.png` — 结果汇总截图

## 判定标准汇总

| 大类 | 子项 | 检查器 | 判定标准 |
|------|------|--------|----------|
| 源码 | 内核 | KernelChecker | 5 步全部通过（仓库在 openEuler 组织 + 版本在主流范围 5.10/6.6） |
| 源码 | 基础中间件 | MiddlewareChecker | libc.so MD5 与 env.yaml 基线**严格相等** |
| 源码 | 其他软件包 | PackageChecker | 社区包占比 **≥ 70%** |
| 构建 | 编译链 | CompilerChecker | gcc 和 glibc MD5 与基线**均严格相等** |
| 构建 | 构建工程 | ProjectChecker | oebuild bitbake 可用 + 目录结构**全部齐全** |
| 构建 | 包列表 | PkgListChecker | 社区包版本一致率 **≥ 70%** |
| 运行时 | C 库运行时 | MiddlewareRuntimeChecker | 检测程序 MD5 一致 + 运行退出码为 0 |
| 运行时 | 基础功能 AT | ATTestChecker | AT 测试用例 **100% 通过** |
| 运行时 | POSIX | PosixChecker | POSIX 测试用例 **100% 通过** |

### 认证结论

| 结果 | 条件 |
|------|------|
| **通过** | 所有检查项均为 PASS |
| **不通过** | 存在任一 FAIL |
| **待整改** | 存在 NONE（未测试）且无 FAIL |
